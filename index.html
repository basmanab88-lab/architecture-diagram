1. Dev loop

Basman opens VS Code. Claude Code reads relevant memory, applies the PRINCIPLES.md rules (auto-injected by a UserPromptSubmit hook based on keywords). Code is written; the Stop hook syncs ~/.claude to GitHub after every session.

2. Source of truth

All operational knowledge is on GitHub. New machine? Clone claude-config and run setup-new-machine.sh — full state restored. No machine is irreplaceable.

3. Auto-deploy

Vercel + Supabase deploy on every push. calendar-skills has a release pipeline that opens auto-PRs in consumer repos, which auto-deploy on merge. Zero-touch end-to-end.

4. Runtime

Users arrive through one of two doors: a browser (Vercel-hosted frontend) or a WhatsApp message (Cloud API webhook). Both land in a Supabase Edge Function, which orchestrates Claude + Google APIs + responses back.

Why public

Supabase Edge Functions run on Deno. Deno imports modules via URL. Private repo URLs need auth headers; a public repo's raw.githubusercontent.com URL is fetchable with no setup. So the repo is public — no secrets allowed in here, ever.

Why a separate repo

So a calendar bug discovered in T-800 doesn't get patched in T-800. It gets patched in calendar-skills, released, and auto-rolls to every consumer. One fix, every product.
